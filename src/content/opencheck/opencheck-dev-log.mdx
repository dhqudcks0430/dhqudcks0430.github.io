## 시작한 이유

OpenCheck는 오픈소스 저장소를 사용할 때 “이 프로젝트를 믿고 설치해도 괜찮을까?”라는 질문에서 시작한 도구입니다.
README 설명, 패키지 설치 스크립트, 의심스러운 명령어, 분석 결과 리포트를 한 흐름으로 묶어 개발자가 빠르게 위험 신호를 확인할 수 있게 만드는 것이 목표입니다.

아직 완성된 제품이라기보다는 학습과 구현을 함께 진행하는 프로젝트입니다. 그래서 이번 글은 결과물 소개보다, 지금까지 어떤 구조를 만들었고 무엇을 배웠는지를 남기는 개발 기록에 가깝습니다.

## 지금까지 만든 것

현재 프로젝트에는 백엔드 분석 흐름, 프론트엔드 결과 화면, 문서화 초안이 들어가 있습니다.

- FastAPI 기반 분석 API 구조
- 저장소 클론 서비스와 테스트 샘플 저장소
- README 파서와 정적 분석기
- 위험 규칙 정의 문서
- 동적 분석 방향 문서
- JSON 리포트 형식 문서
- React 기반 분석 결과 요약 화면

GitHub 업로드는 로컬 커밋까지 완료했고, 마지막 `git push` 단계에서 인증 설정이 필요합니다. 이 부분은 Personal Access Token 또는 GitHub CLI 로그인을 설정한 뒤 이어갈 예정입니다.

## 분석 규칙 생각하기

초기 정적 분석은 “위험할 수 있는 신호”를 먼저 찾는 방식으로 잡았습니다.

예를 들어 다음과 같은 패턴을 확인합니다.

```bash
curl http://example.com/script.sh | sh
chmod +x install.sh
rm -rf /tmp/some-path
```

이런 명령어가 항상 악성이라는 뜻은 아닙니다. 하지만 설치 과정에서 원격 스크립트를 바로 실행하거나, 파일 권한을 바꾸거나, 삭제 명령을 수행한다면 사용자가 한 번 더 살펴볼 필요가 있습니다.

## 리포트 형식

분석 결과는 사람이 읽기 쉬운 요약과 기계가 처리하기 쉬운 JSON을 같이 생각하고 있습니다.
리포트에는 저장소 기본 정보, 발견된 위험 신호, 점수, 추천 확인 사항을 넣는 방향입니다.

```json
{
  "repository": "sample/risky-project",
  "riskLevel": "high",
  "findings": [
    {
      "file": "install.sh",
      "rule": "remote-script-execution",
      "severity": "high"
    }
  ]
}
```

이 형식을 먼저 안정화하면, 프론트엔드 화면과 CLI 출력도 같은 데이터를 기반으로 만들 수 있습니다.

## 배운 점

이번 작업을 하면서 `.venv`의 역할도 다시 정리했습니다.
Python 프로젝트마다 필요한 패키지 버전이 다르기 때문에, `.venv`는 프로젝트 전용 Python 실행 환경을 분리해 주는 폴더입니다.

또 하나 배운 점은 도구를 만들 때 분석 로직만큼 문서가 중요하다는 점입니다. 위험 규칙을 왜 그렇게 판단하는지 설명하지 못하면, 사용자는 결과를 믿기 어렵습니다.
