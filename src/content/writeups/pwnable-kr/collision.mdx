## Overview

`collision` looks like a crypto challenge only because of the name. The actual check is much simpler: the program takes a 20-byte argument, casts it to five 32-bit integers, and checks whether their sum is `0x21DD09EC`. Once that is clear, the solve is just controlled little-endian packing.

## Challenge Snapshot

| Field | Value |
| --- | --- |
| Primitive | 20-byte argv value reinterpreted as five integers |
| Core Logic | `a[0] + a[1] + a[2] + a[3] + a[4] == 0x21DD09EC` |
| Goal | Pack five 32-bit chunks whose sum hits the target |

## Environment & Tools

The solve is tiny enough that Python is all you need. The only easy mistake is endianness.

```python
from struct import pack

payload = pack('<I', 0x06C5CEC8) * 4 + pack('<I', 0x06C5CECC)
```

## Recon

Two source details matter:

1. `strlen(argv[1])` must be `20`
2. the password is cast to `int *` and summed across five words

That means this is not a real hash collision. It is a five-word integer-sum problem with a hard-coded target.

## Vulnerability Analysis

The target value is:

```text
0x21DD09EC
```

One common valid split is:

```text
0x06C5CEC8 * 4 + 0x06C5CECC = 0x21DD09EC
```

So the only real job is to deliver those five integers as raw little-endian bytes.

## Exploitation Steps

I used four equal words and one remainder word, then emitted them as raw bytes.

```bash
python -c 'print "\xc8\xce\xc5\x06"*4 + "\xcc\xce\xc5\x06"' | xxd
```

When used as the 20-byte argv payload, those words satisfy the check without brute force.

## Payload/Technique Breakdown

This challenge becomes “real” once you stop thinking in strings and start thinking in machine words:

- the program wants exactly 20 bytes,
- those bytes are treated as 5 little-endian integers,
- the sum must equal `0x21DD09EC`.

## Result

With the bytes packed in little-endian order, the check passes immediately and the binary prints the success path.

## Lessons Learned

- The challenge name is more misleading than the code.
- Integer interpretation details matter more than “crypto” intuition here.
- For argv-based puzzles, raw byte control is often the whole challenge.
